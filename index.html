What is Pro Scanner?

Pro Scanner is a local-first web security assessment platform. The browser UI communicates exclusively with a Python backend running on your machine at localhost:8765. No scan data, credentials, or target information is transmitted to external servers.

The platform automates the most common categories of web vulnerability detection: injection flaws, TLS/certificate misconfigurations, API exposure, and network surface mapping. Results are heuristic signals — always validate manually before treating findings as confirmed vulnerabilities.

Setup & Backend Connection

Before using any scan module, the Python backend must be running. The status indicator in the bottom-left sidebar shows real-time connection state.

Starting the backend
$ python backend.py
# Server starts on http://localhost:8765
✓
Backend Online
Status dot is green and pulsing. All scan modules are available. Results stored in the local database.
✗
Backend Offline
Status dot is red. Start python backend.py in your terminal. Scans will fail until resolved.
Python requirements: Python 3.8+, with requests, flask, and flask-cors installed. Run pip install -r requirements.txt if packages are missing.

Recommended Workflow

For systematic assessments, follow this sequence. Each step builds context for the next.

1
Scope & Authorization
Confirm written permission for the target domain(s). Document scope boundaries. Never test production systems without explicit consent.
2
SSL / TLS First Pass
Run the SSL Checker on the target hostname. Certificate issues and weak TLS versions affect all subsequent HTTP traffic security.
3
Port Scanner — Network Surface
Map exposed ports and identify running services. Open database ports or admin panels are often high-priority findings before web testing begins.
4
Vulnerability Scanner — Web Application
Paste parameterized URLs and run SQLi, XSS, and path traversal tests. Select test categories relevant to the application type.
5
API Scanner — REST/GraphQL Endpoints
Probe API endpoints for open access, CORS misconfigurations, auth bypasses, and sensitive data exposure patterns.
6
Report & Remediate
Generate a PDF report, review fix cards for each finding, use Custom Tests to re-verify after remediation is applied.

Module Reference

Click any module card to expand detailed usage instructions, configuration options, and result interpretation guidance.

🔍
Vulnerability Scanner
SQL Injection · XSS · Path Traversal · SSRF · Header Injection
▼ Expand
The Vulnerability Scanner sends crafted HTTP requests to URLs containing query parameters and analyzes server responses for signatures that indicate injection or traversal vulnerabilities. Each parameter in the URL is tested independently across all enabled test categories.
1
Ensure the backend is running — the sidebar status dot must be green. Start it with python backend.py if needed.
2
Navigate to Scanner. In the Target URL field, paste a full URL with query parameters, e.g. http://testphp.vulnweb.com/artists.php?artist=1. The scanner tests each parameter it detects.
3
Select which vulnerability categories to test in the Test Modules panel. Use All for a comprehensive scan, or pick specific categories (SQLi, XSS, etc.) for targeted testing. Each chip shows a payload count.
4
Click RUN SCAN. The live log streams real-time progress — each entry shows timestamp, log level, and the test being executed. VULN entries indicate a potential finding.
5
When complete, a verdict banner summarizes the result: VULNERABLE (red) or SAFE (green), with a finding count. The result table below lists every test, its payload, response time, and risk classification.
6
Expand Fix Cards for any VULN findings to view remediation code examples, OWASP references, and configuration guidance specific to each vulnerability type.
7
Navigate to Reports to generate a PDF of the full findings, suitable for development teams or security assessment documentation.
Result Interpretation: Risk levels: critical authentication bypass or full data access → high significant exposure → medium reflected XSS → low hardening issues. Always manually verify VULN findings — false positives can occur with WAFs and custom error pages.
🔒
SSL / TLS Checker
Certificate Validity · TLS Version · HSTS · Redirect Chain
▼ Expand
The SSL/TLS Checker validates the full certificate chain, negotiated TLS protocol versions, HSTS header presence and configuration, and whether HTTP traffic is properly redirected to HTTPS. These checks are foundational — a weak TLS configuration undermines all other application-layer controls.
1
Navigate to SSL Check in the sidebar.
2
Enter a hostname or full URL. Both formats are accepted: example.com or https://example.com. Do not include URL paths.
3
Click CHECK SSL. Results appear organized into six check categories: certificate validity, expiry date, TLS version, HSTS, HTTPS redirect, and cipher strength.
4
Each check shows a status icon: ✓ green (pass), ⚠ amber (warning), or ✗ red (fail). Failures include descriptions of the specific issue.
5
Expand Fix Cards for each red finding to view server-specific configuration snippets (nginx, Apache, IIS) to resolve the issue.
Key checks: TLS 1.0 and 1.1 are deprecated — disable them. HSTS with max-age ≥ 31536000 is recommended. Certificates expiring within 30 days trigger a warning even if currently valid.
⚡
API Security Scanner
REST/JSON Probing · CORS · Auth Bypass · Data Exposure · GraphQL
▼ Expand
The API Scanner probes REST and GraphQL endpoints using GET and POST methods. It detects unauthorized access to sensitive paths, CORS policy misconfigurations, missing authentication enforcement, and response body patterns that suggest data over-exposure.
1
Navigate to API Scanner in the sidebar.
2
Enter the API base URL, e.g. https://api.example.com/v1. This is the root from which paths will be appended.
3
Optionally enter specific paths in the textarea, one per line (e.g. /users, /admin). Leave blank to use the built-in probe list of 40+ common API paths.
4
Click SCAN API. The scanner dispatches requests to each path and evaluates responses for HTTP status codes, response headers, CORS policies, and body content patterns.
5
Results show a Security Score (0-100), severity breakdown, summary of findings, and an endpoint-by-endpoint table with method, status, and flagged issues.
6
Results are heuristic — a 200 response on /api/admin is a signal, not a confirmed bypass. Always verify manually.
CORS findings: A wildcard Access-Control-Allow-Origin: * on sensitive data endpoints is a genuine vulnerability. On public data endpoints it is acceptable.
🌎
Port Scanner
15 Common Ports · Service ID · Firewall Exposure · Risk Rating
▼ Expand
The Port Scanner resolves the target hostname via DNS and then attempts TCP connections to 15 high-value ports. Open ports are identified and tagged with their associated service and a risk classification based on service type and exposure context.
1
Navigate to Port Scan in the sidebar.
2
Enter a hostname or IP address, e.g. scanme.nmap.org. Full URLs are also accepted — the scanner extracts the hostname automatically.
3
Click SCAN PORTS. DNS resolution happens first, then TCP probes are sent in parallel. A progress bar tracks completion across the 15 ports.
4
Results show each port with status: OPEN (connection accepted), CLOSED (refused), or FILTERED (timeout). Open ports include service name and risk rating.
5
High-risk open ports (databases on 3306/5432, SMB on 445, RDP on 3389) generate fix recommendations with firewall rules and hardening guidance.
Legal reminder: Port scanning without authorization is illegal in many jurisdictions. Only scan systems you own or have explicit written permission to test. Use scanme.nmap.org for safe practice.
⚙
Custom Tests
Save Checks · Re-run After Fixes · GET & POST · Expected Status
▼ Expand
Custom Tests let you define, save, and re-run arbitrary HTTP requests. Useful for verifying that fixes have been applied, testing non-standard endpoints, or building a regression suite that runs before and after deployments.
1
Navigate to Custom Tests in the sidebar.
2
Fill in the test form: give it a descriptive Name, the full Target URL, select the HTTP Method (GET or POST), optionally provide a request Payload for POST bodies, and set the Expected Status code (default 200).
3
Click SAVE TEST. The test is persisted in the backend database and appears in the saved tests list below.
4
Click ▶ Run next to any saved test to execute it immediately. Results show HTTP status, response time, and whether flagged patterns were detected in the response body.
5
A test passes when the actual HTTP status matches your expected status. Use this to verify that a previously open endpoint now correctly returns 401 or 403 after authentication controls are applied.

Frequently Asked Questions

Why does the backend show as offline? ▼
The frontend pings localhost:8765/api/scans on load. If it times out, the backend process is not running. Open a terminal, navigate to the project directory, and run python backend.py. Check that no other process is using port 8765.
Are scan results stored remotely? ▼
No. All results are stored in a local SQLite database managed by the Python backend on your machine. The frontend only communicates with localhost:8765. No data is sent to any external server by Pro Scanner itself.
Why do scans show false positives? ▼
The scanner uses heuristic pattern matching on server responses. WAFs, custom error pages, and applications that return similar responses regardless of input can all produce false positives. Always manually verify findings: confirm that the injection point actually affects application behavior and that returned data is genuinely sensitive.
How do I scan a URL without query parameters? ▼
The Vulnerability Scanner requires URLs with query parameters (e.g. ?id=1). For pages without parameters, use Custom Tests to craft POST requests with a body payload, or use the API Scanner for REST endpoints. You can also manually add a dummy parameter to explore scanner behavior.
What is the difference between risk levels? ▼
Risk levels reflect both the severity of the vulnerability class and the confidence of the finding:
CriticalAuthentication bypass, direct database access, or remote code execution potential. Address immediately.
HighSignificant data exposure or privilege escalation risk. Address within the current sprint.
MediumReflected XSS, CORS issues, or configuration weaknesses. Address in the next release cycle.
LowInformation disclosure, missing security headers, or hardening recommendations. Track and remediate.
